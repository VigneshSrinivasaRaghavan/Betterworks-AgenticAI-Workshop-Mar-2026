    # Memory Types Explained
    
    ## What We'll Cover
    
    Understand short-term vs long-term memory, when to use each, and how both work together in our LangGraph agents
    
    ---
    
    ## The Problem: Stateless Agents
    
    Current agents (without memory):
    
    ```
    User: "Generate login test cases"
    Agent: <generates 5 test cases>
    
    User: "Add edge cases to those tests"
    Agent: "Which tests? I don't remember what we just discussed."
    ```
    
    Issue: Agent forgets immediately after each run
    
    ---
    
    ```
    User: Uploads database error log
    Agent: "Connection pool exhausted - fix by restarting service-b"
    
    (Next day, same issue appears)
    
    User: Uploads similar error log
    Agent: "I have no idea if we've seen this before."
    ```
    
    Issue: Agent has no memory of past sessions
    
    ---
    
    ## The Solution: Two Types of Memory
    
    ---
    
    ## Type 1 — Short-term Memory (Within Session)
    
    What it is:
    
    - Remembers the current conversation while the agent is running
    - Lives in LangGraph state, managed by MemorySaver
    - Tied to a session using a thread_id
    - Gone when the process exits (in-memory by default)
    - Like human working memory — active only while you're thinking
    
    How it works in LangGraph:
    
    ```python
    from langgraph.checkpoint.memory import MemorySaver
    
    checkpointer = MemorySaver()
    graph = workflow.compile(checkpointer=checkpointer)
    
    # Each call with the same thread_id shares memory
    config = {"configurable": {"thread_id": "session-1"}}
    
    graph.invoke({"requirement": "login tests"}, config=config)
    graph.invoke({"requirement": "add 2FA tests"}, config=config)
    # Second call sees everything from first call
    ```
    
    What the agent sees on the second call:
    
    ```
    conversation_history: [
      {"role": "user",  "content": "Generate login tests"},
      {"role": "agent", "content": "Generated 5 login tests: TC001, TC002..."},
      {"role": "user",  "content": "Add 2FA tests"}
    ]
    ```
    
    The LLM uses this history to give a contextual response.
    
    Example:
    
    ```
    User: "Generate login API tests"
    Agent: <generates 5 tests>
    
    User: "Add edge cases to those"
    Agent: "I'll add edge cases to the 5 login tests we just created..."
    
    User: "You missed the logout scenario"
    Agent: "You're right — adding logout to the existing 6 tests..."
    ```
    
    Use when:
    
    - Following up on what the agent just generated
    - Refining output in the same session
    - Multi-turn conversation with the agent
    
    ---
    
    ## Type 2 — Long-term Memory (Across Sessions)
    
    What it is:
    
    - Remembers past sessions
    - Stored in ChromaDB as vector embeddings
    - Persists on disk — survives process restarts
    - Searched using semantic similarity (same ChromaDB you already know)
    - Saved only when user confirms the output is good
    
    Important — Two Separate ChromaDB Collections:
    
    ```
    ChromaDB on disk
    ├── data/vector_store/          ← RAG Collection (static)
    │   Purpose: company guidelines, testing standards, best practices
    │   Built once using build_vector_store()
    │   Never changes during agent runs
    │
    └── data/memory/testcase_memory/   ← LTM Collection (grows over time)
        Purpose: past test cases generated by the agent
        Grows with every confirmed session
        Searched to find patterns from previous runs
    ```
    
    The RAG collection and LTM collection are completely separate. Both are searched at the start of each run and both results go into the LLM prompt.
    
    How LTM is saved (with user confirmation in the driver):
    
    ```
    Agent generates test cases
    Driver shows them to user
    Driver asks: "Approve and save to memory? (y/n)"
      If y → PersistentMemory.store_interaction(...) → written to ChromaDB
      If n → user gives more instructions → agent refines → ask again
    ```
    
    What the agent retrieves on Session 2:
    
    ```
    Session 1 (saved login tests to LTM)
    
    Session 2 — User: "Generate registration tests"
    Agent searches LTM: "test patterns for API tests"
    Finds: "Session 1 — login tests with 2FA, rate limiting, P1/P2 priorities"
    LLM uses this to generate consistent, pattern-aware registration tests
    ```
    
    Use when:
    
    - Referencing test patterns from past sessions
    - Recalling past incidents or fixes
    - Building on previous work across different days
    - Maintaining consistency across sessions
    
    ---
    
    ## Comparison
    
    Short-term Memory:
    
    - Duration: Current session only
    - Storage: LangGraph state (MemorySaver in-memory)
    - Reset: When process exits
    - Speed: Instant (already in state)
    - Use case: Follow-ups, refinements within one run
    
    Long-term Memory:
    
    - Duration: Forever (until deleted)
    - Storage: ChromaDB on disk (separate collection from RAG)
    - Reset: Never (or on user request)
    - Speed: Fast (vector similarity search)
    - Use case: Cross-session patterns, past incidents, consistency

    ## Real-World Examples
    
    ### Example 1: TestCase Generator Agent
    
    Session 1 — Short-term memory in action (same CLI run):
    
    ```
    User: "Generate login API tests"
    Agent: <generates 5 tests: TC001-TC005>
    
    User: "Add 2 more for the logout flow"
    Agent: "Adding to the 5 login tests we created — TC006, TC007 for logout..."
    
    User: "You missed rate limiting"
    Agent: "Adding TC008 for rate limiting to the 7 tests we have so far..."
    
    Driver: "Approve and save to memory? (y/n)" → y
    → 8 tests saved to LTM (ChromaDB testcase_memory collection)
    ```
    
    Session 2 — Long-term memory in action (next day, new process):
    
    ```
    User: "Generate registration API tests"
    Agent searches LTM → finds login tests from Session 1
    Agent: "Based on your previous login tests (2FA, rate limiting, logout patterns),
            I'll apply the same structure to registration..."
    <generates consistent, pattern-aware registration tests>
    ```
    
    ---
    
    ### Example 2: Log Analyzer Agent
    
    Session 1 — Short-term memory in action:
    
    ```
    User: Uploads database error log
    Agent: "Connection pool exhausted at 09:17. Root cause: Microservice-B leak"
    
    User: "What about the timeout at 09:20?"
    Agent: "In the log we just analyzed, the timeout is a downstream effect
            of the connection pool issue we identified..."
    ```
    
    Session 2 — Long-term memory in action:
    
    ```
    User: Uploads similar error log (next day)
    Agent searches LTM → finds Session 1's incident
    Agent: "This matches the connection pool issue from yesterday.
            Last time, restarting service-b resolved it in 10 minutes..."
    ```

    ## Complete Flow — Both Memories Together
    
    ```
    SESSION 1
    ──────────────────────────────────────────────
    Run agent (thread_id = "session-1")
    
    Step 1 — load_memories node:
      RAG search   → company testing guidelines (static collection)
      LTM search   → past test patterns (empty first time)
    
    Step 2 — generate_tests node:
      LLM uses: RAG guidelines + empty LTM + requirement
      Output: 5 test cases
    
    Step 3 — Driver shows output, asks for follow-up or approval:
      User: "Add 2 more for edge cases"
      → graph.invoke() again with same thread_id
      → MemorySaver restores conversation_history
      → LLM sees previous 5 tests, adds 2 more
    
      User: "Looks good, save"
      → Driver writes 7 tests to LTM (ChromaDB testcase_memory)
    
    SESSION 2 (next day — new process, new thread_id)
    ──────────────────────────────────────────────
    Run agent (thread_id = "session-2")
    
    Step 1 — load_memories node:
      RAG search   → guidelines (same as always)
      LTM search   → finds Session 1's 7 tests ✓
    
    Step 2 — generate_tests node:
      LLM uses: RAG guidelines + Session 1 patterns + new requirement
      Output: builds on Session 1's patterns, consistent style
    
    Step 3 — Driver loop continues...
      User refines → approves → saved to LTM again
      LTM grows with each session
    ```
    
    ---
    
    ## When to Use Each
    
    Use Short-term Memory when:
    
    - User says "add more to those" or "change that priority"
    - User is refining what the agent just produced
    - You need multi-turn conversation within one run
    - Context needs to carry across 2-3 follow-up messages
    
    Use Long-term Memory when:
    
    - You want the agent to recall what it did last week
    - You want consistent test patterns or incident responses across sessions
    - You want to avoid repeating work
    - You want the agent to get smarter over time
    
    Use Both Together when:
    
    - Building production-grade agents
    - Teaching agent + refinement loop + memory continuity
